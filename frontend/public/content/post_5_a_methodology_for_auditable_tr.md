### 1. 추상적인 선언과 파편화된 기술 지표의 이분법
현재 AI 거버넌스(특히 **유럽 연합 AI법(EU AI Act)** 등 고위험 AI 규제 대응)는 두 가지 극단적인 방향으로 나뉘어 있습니다.
* **거버넌스적 접근(High-Level):** ALTAI, NIST AI RMF 등은 신뢰성 요건(공정성, 안전성, 투명성 등)을 제안하지만, 이를 배포 후 매 시간 변화하는 시스템 상에서 어떻게 지속적으로 점검할지에 대한 정량적 방법론이 없습니다.
* **엔지니어링적 접근(Low-Level):** MLOps 진영에서는 성능 지표나 단순 데이터 드리프트를 추적하지만, 그것이 거버넌스 관점에서 **"지금 시스템이 '수용 가능한(Acceptable)' 수준인가?"** 혹은 **"당장 서비스를 멈춰야 하는 '위험한(Risky)' 상태인가?"**에 대한 거버넌스적 판단으로 이어지지 못합니다.

### 2. '중대한 변경(Substantial Modification)' 감지의 한계
EU AI Act 제3조 및 제43조에 따르면, 고위험 AI에 성능 변화가 발생하여 적합성 평가 기준을 벗어나는 **중대한 변경**이 있을 시 적합성 재평가를 받아야 합니다. 하지만 기존의 정적인 모델 카드(Model Cards)나 개발 시점의 정적 보고서로는 운영 중인 AI가 어느 순간 적합성 임계점을 넘었는지 실시간으로 감지하고 문서화하기 어렵습니다.

## 💡 [핵심 기술 및 아키텍처]
> 💡 **한 줄 요약:** 정형화된 프로토콜과 의사결정나무(Decision Tree)를 결합하여 감사가 가능한 동적 신뢰성 평가 모형을 정의합니다.

이 논문은 복잡한 거버넌스 요구사항을 수학적으로 계량화하여 컴퓨터가 규칙을 학습할 수 있도록 정밀한 아키텍처를 제시합니다.

### 1. 신뢰성 프로토콜과 프로파일의 수학적 정의

인공지능 시스템 $A$의 의도된 목적 $I_A$와 사용 맥락 $C_A$가 주어졌을 때, 시스템의 신뢰성 상태는 다음과 같은 계층 구조로 수학적 모델링을 거칩니다.

```
[ 신뢰성 프로파일 수집 및 연산 구조 ]

  Raw Metrics (q_ij)      Dimension Scores (Q_i)       Trustworthiness Profile (Q_A)
┌────────────────────┐    ┌─────────────────────┐    ┌───────────────────────────────┐
│ Accuracy/F1        ├───►│ Robustness   [0.82] ├───►│                               │
├────────────────────┤    ├─────────────────────┤    │ QA(t) =                       │
│ Perturbation Test  ├───►│ Fairness     [0.91] ├───►│ [ 0.82, 0.91, 0.78, ... ]     │
├────────────────────┤    ├─────────────────────┤    │ (시간 t에서의 다차원 벡터)      │
│ Demog. Parity Gap  ├───►│ Explainability [0.78]├───►│                               │
└────────────────────┘    └─────────────────────┘    └───────────────────────────────┘
```

#### ① 신뢰성 프로토콜 (Trustworthiness Protocol)
해당 도메인에 부합하는 $n$개의 신뢰성 차원(Robustness, Fairness, Transparency 등)의 집합을 정의합니다.
$$P_A = \{d_1, d_2, \dots, d_n\}$$

#### ② 차원별 하위 메트릭 및 벡터 ($q_i^A(t)$)
각 차원 $d_i$는 $k(i)$개의 구체적인 하위 성능 지표로 조작화(Operationalization)되며, 모든 값은 $[0, 1]$ 범위로 정규화됩니다.
$$q_i^A(t) = \left( q_{i,1}^A(t), q_{i,2}^A(t), \dots, q_{i,k(i)}^A(t) \right) \in [0, 1]^{k(i)}$$

#### ③ 차원 통합 스코어 ($Q_i^A(t)$)와 프로파일 ($Q_A(t)$)
차원별 집계 함수 $a_i: [0, 1]^{k(i)} \to [0, 1]$를 거쳐 단일 점수를 낸 후, 이를 한데 모아 해당 시점 $t$의 **신뢰성 프로파일 벡터**를 생성합니다.
$$Q_i^A(t) := a_i(q_i^A(t))$$
$$Q_A(t) := \left( Q_1^A(t), Q_2^A(t), \dots, Q_n^A(t) \right) \in [0, 1]^n$$

### 2. 경험적 신뢰성 규칙 학습 (Empirical Trustworthiness-Level Learning)

사전에 전문가 그룹의 검토나 시뮬레이션을 통해 확보한 레이블된 데이터셋 $D_N = \{(Q_A(t_i), Y_i)\}_{i=1}^N$이 존재합니다. 여기서 $Y_i \in \{1, 2, \dots, K\}$는 사전에 정의된 신뢰성 등급(예: $1$=위험, $2$=수용가능, $3$=양호, $4$=최우수)입니다.

목표는 프로파일 영역 $[0, 1]^n$을 축에 평행하게 분할하는 **해석 가능한 계단식 함수(Interpretable Stepwise Function)** $h$를 학습하는 것입니다. 논문에서는 감사(Audit) 과정에서 직관적인 규칙(Rule-based) 도출을 위해 **의사결정나무(Decision Tree)** 모형 클래스 $\mathcal{H}_{\text{tree}}$를 활용해 최적화 문제를 풉니다.

$$\hat{T}_{A,N} = \arg\min_{h \in \mathcal{H}_{\text{tree}}} \left( \frac{1}{N} \sum_{i=1}^N \ell(h(Q_A(t_i)), Y_i) + \lambda \Omega(h) \right)$$

* 여기서 $\ell$은 예측 오류에 대한 손실 함수이며, $\Omega(h)$는 트리의 깊이(Depth)나 리프 노드 수(Leaves)를 제한하는 복잡도 패널티 함수입니다.
* $\lambda$는 규제 완화와 단순함 간의 균형을 조정하는 하이퍼파라미터입니다.

### 3. 실시간 감사 핵심 지표 (Auditing Diagnostics)

모델 배포 후 실시간 모니터링 단계에서 핵심적으로 활용되는 2가지 신규 지표를 제안합니다.

```
[ Boundary Margin과 Profile Drift의 개념적 시각화 ]

                   (Level 3: 양호)
         ┌───────────────────────────────┐
         │                               │
         │       ● Q_A(t-h) [과거]        │
         │       │                       │
         │       │ Profile Drift D_A(t)  │
         │       ▼                       │
         │       ● Q_A(t)   [현재]        │
         │       │                       │
         │───────┼───────────────────────│ ◄─── Level Boundary (의사결정 경계면)
         │       │ Boundary Margin κ_A(t)│
         └───────┴───────────────────────┘
                   (Level 2: 주의)
```

#### ① 경계 마진 (Boundary Margin)
현재 신뢰성 프로파일 $Q_A(t)$가 의사결정 경계면 $\partial \mathcal{C}$에 얼마나 인접해 있는지를 나타냅니다. 이 값이 작을수록 시스템이 조만간 하위 신뢰 등급으로 추락할 위험이 있는 **'공간적 취약성(Spatial Fragility)'** 상태에 직면했음을 뜻합니다.
$$\hat{\kappa}_{A,N}(t) := \text{dist}\left( Q_A(t), \partial \mathcal{C}_{\hat{\tau}_{A,N}(t)} \right)$$

#### ② 프로파일 드리프트 (Profile Drift)
과거 특정 윈도우 크기 $h$ 이전의 프로파일 상태와 비교하여 신뢰성 축 상에서 시스템의 변화 속도를 의미하는 **'시간적 변동 속도(Temporal Velocity)'**를 측정합니다.
$$D_A(t; h) := \frac{1}{\sqrt{n}} \| Q_A(t) - Q_A(t-h) \|_2$$

## 📊 [실험 결과 분석]
> 💡 **한 줄 요약:** 추천 시스템, 대출 평가, 의료 진단 등 3가지 가상 시나리오 시뮬레이션을 통해 방법론의 실효성을 숫자로 검증했습니다.

논문은 통제된 합성 시뮬레이션(Synthetic Trace) 환경 하에 3가지 실험을 설계하고 성능을 분석했습니다.

### 🧪 [실험 1] 추천 시스템 vs 의료 의사결정 지원 시스템 (동적 모니터링의 주기성 차이)
* **목적:** 빈번한 데이터가 쌓이는 밀접 모니터링 환경(추천 시스템, 720일)과 실제 환자 결과 확인 등 피드백 루프가 느려 드문드문 모니터링되는 환경(의료 AI, 48개월)에서의 거버넌스 룰 작동 여부를 검증합니다.
* **사용 지표:** 정확도(Accuracy) 및 강건성(Robustness).
* **예측 성능 데이터:**

| 시스템 시나리오 | 학습 MAE / 평가 MAE | 정확도 (Train / Test) | 균형 정확도 (Train / Test) | Macro-F1 (Train / Test) |
| :--- | :--- | :--- | :--- | :--- |
| **추천 시스템 (Dense)** | 0.00 / 0.00 | **1.00 / 1.00** | **1.00 / 1.00** | **1.00 / 1.00** |
| **의료 AI (Sparse)** | 0.05 / 0.00 | **0.95 / 1.00** | **0.75 / 1.00** | **0.73 / 1.00** |

* **분석 결과:** 빈번하고 노이즈가 적은 추천 시스템 환경에서는 가상의 전문가 룰을 완벽하게 재현했습니다. 의료 AI와 같이 데이터가 희소한 환경에서도 학습 데이터에서 95%, 평가 데이터에서 100%의 정확도를 유지하며, 소수의 전문가 등급을 강건하게 모사하는 3단계 단순화 규칙 학습에 성공했습니다.

### 🧪 [실험 2] 금융 대출 승인 시스템 (비동기적 다차원 지표 환경)
* **목적:** 성능(AUROC), 설명력(Reason-Code Stability), 강건성(Stress-test), 공정성(Equal Opportunity Gap) 등 총 4개 차원의 지표가 서로 다른 주기(4주, 12주, 16주)로 비동기적으로 들어올 때의 거버넌스 안정성을 평가합니다. (총 90회 리뷰 세션 진행)
* **예측 성능 데이터:**

| 학습 MAE / 평가 MAE | 정확도 (Train / Test) | 균형 정확도 (Train / Test) | Macro-F1 (Train / Test) |
| :--- | :--- | :--- | :--- |
| **0.11 / 0.17** | **0.96 / 0.83** | **0.75 / 0.50** | **0.71 / 0.45** |

* **분석 결과:** 데이터 불균형(대부분 정상 레벨 4에 안착)이 심한 환경이었으나, 학습 단계에서 96%의 정확도로 신뢰 등급을 추적했습니다. 다만, 테스트 단계에서 F1-score가 0.45로 하락했는데, 이는 신뢰성이 급격히 무너지는 저신뢰 상태(Level 1, 2) 데이터의 노출이 극단적으로 부족한 '희소 레이블 문제' 때문입니다. 이는 머신러닝 모델이 거버넌스 규칙을 "단순화(Compression)"해 표현하게 됨을 보여주며 실무적으로 주의해야 할 포인트를 제시합니다.

### 🧪 [실험 3] 동일 의료 AI 모델의 다른 병원 배포 비교 (도메인 차이에 따른 감사 분석)
* **목적:** 완전히 동일하게 개발된 의료 AI 소프트웨어를 안정적인 환경의 Hospital Alpha와 환자군 변화 및 장비 교체가 빈번한 Hospital Beta 두 곳에 각각 배포했을 때의 거버넌스 의사결정을 비교합니다. (6개 차원 지표 활용)
* **예측 성능 데이터:**

| 대상 기관 | 사용 규칙 종류 | 학습 MAE / 평가 MAE | 정확도 (Train / Test) | 균형 정확도 (Train / Test) | Macro-F1 (Train / Test) |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **Hospital Alpha** | 전문가 정적 룰 (유지) | N/A | N/A | N/A | N/A |
| **Hospital Beta** | 데이터 기반 학습 룰 | **0.11 / 0.50** | **0.89 / 0.50** | **0.75 / 0.50** | **0.68 / 0.33** |

* **분석 결과:**
  1. **Hospital Alpha:** 모델 성능이 배포 기간 전반에 걸쳐 항상 최적 상태(Level 4)를 유지하여 데이터에 변동이 없었습니다. 이 경우 프레임워크는 머신러닝 학습 모델을 적용하는 대신 **"기존 전문가 룰 유지"** 결정을 내림으로써 오버피팅과 거버넌스 리소스 낭비를 막았습니다.
  2. **Hospital Beta:** 시스템 성능 하락 및 장비 쇼크가 빈번하여 4개 신뢰 등급을 수시로 오갔습니다. 여기서는 학습 모델이 도입되었으나 배포 이후 테스트 데이터셋에서의 성능 저하(정확도 50%, Macro-F1 0.33)가 감지되었습니다. 이는 프레임워크가 단순히 모델 신뢰도를 분류하는 것을 넘어, **"현재 병원의 환경 변화가 너무 심하므로 신뢰성 평가지표(Protocol) 자체를 개정하거나 전면적인 인간 재평가가 필요한 시점이다"**라는 정량적 신호를 규제 당국에 제공하는 역할을 함을 의미합니다.

### 🏢 엔터프라이즈 AI 거버넌스 실무 적용 시나리오
> **"국내 시중은행의 LLM 기반 대고객 금융 어드바이저 시스템 거버넌스 설계"**

만약 여러분이 시중은행에서 금융 규제 요건을 충족하면서 생성형 AI 비서를 안정적으로 운영해야 하는 거버넌스 책임자(CAIO) 혹은 데이터 엔지니어라면, 이 논문의 프레임워크를 아래와 같이 제품 생명 주기(AI Lifecycle)에 내재화할 수 있습니다.

```
[ 시중은행 LLM 서비스의 3단계 거버넌스 구현 로드맵 ]

  Phase 1: 디자인-타임          Phase 2: 학습 및 검증             Phase 3: 포스트-배포 모니터링
┌─────────────────────────┐    ┌─────────────────────────┐    ┌─────────────────────────┐
│ - 4대 차원 메트릭 선정     │    │ - 샌드박스 파일럿 운영     │    │ - 2주 단위 프로파일 집계   │
│   (환각율, 편향도 등)     ├───►│ - 의사결정나무 모델 학습    ├───►│ - 신뢰도 저하 알람 트리거 │
│ - 전문가 등급 레이블 정의   │    │ - 거버넌스 아티팩트 문서화  │    │ - '중대 변경' 필요성 판단 │
└─────────────────────────┘    └─────────────────────────┘    └─────────────────────────┘
```

#### Step 1. [Phase 1] 신뢰성 프로토콜 디자인-타임 (Design-Time) 세팅
* **차원 정의:** 금융 서비스에 필수적인 4대 신뢰성 차원을 선정합니다.
  1. **성능($d_1$):** 금융 정보 질문에 대한 환각(Hallucination) 방지율
  2. **강건성($d_2$):** 악의적인 프롬프트 인젝션(Adversarial Prompt) 방어 성공률
  3. **공정성($d_3$):** 성별/연령별 우대 금리 추천 편향도 (Disparate Impact)
  4. **인간 개입($d_4$):** AI 답변에 대해 전문 상담원이 개입하여 차단하거나 가로챈 비율 (Override Rate)
* **전문가 레이블링:** 금융/법률/윤리 전문가들이 모여 샌드박스 테스트 단계의 다양한 프로파일 결과물에 대해 $1$(위험)부터 $4$(안전)까지의 등급 레이블을 부여합니다.

#### Step 2. [Phase 2] 경량 머신러닝 학습 및 문서화 (Artifact)
* 수집된 샌드박스 데이터를 바탕으로 얕은 의사결정나무(정형화 규칙)를 학습합니다.
* 학습된 규칙 예시: 
  * `IF 환각방지율 < 85% OR 금리편향도 > 0.20 THEN Level 1 (위험, 서비스 중단 및 롤백)`
  * `IF 환각방지율 >= 95% AND 인젝션방어율 >= 90% AND 상담원개입률 < 5% THEN Level 4 (안전, 자율 운영)`
* 본 기술 블로그의 **Table 3(최소 거버넌스 아티팩트)** 규격에 맞추어 적합성 문서(Conformity Documentation)를 작성해 사내 아카이브 및 규제 당국(AIA 규제기관)에 제출합니다.

#### Step 3. [Phase 3] 배포 후(Post-deployment) 지속적 모니터링 및 재평가
* 배포 후 MLOps 파이프라인(예: MLflow, Prometheus 등)과 연동하여 **2주 단위**로 최신 프로파일 $Q_A(t)$을 집계합니다.
* 매주 계측되는 **Boundary Margin**이 $0$에 가깝거나 **Profile Drift**가 임계값 $0.15$를 초과할 경우, 슬랙(Slack) 얼럿을 트리거하여 내부 위험 관리 위원회를 소집합니다.
* 시스템이 Level 2(주의)로 떨어질 경우 자동 재학습 파이프라인을 작동하고, 그럼에도 개선되지 않고 Level 1(위험)로 하락할 경우 "중대한 변경" 상황으로 간주하여 신속하게 서비스를 일시 중지하고 재인증 절차를 밟습니다.
