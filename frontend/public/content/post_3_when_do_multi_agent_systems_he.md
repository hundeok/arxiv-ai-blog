# 에이전트를 무작정 쪼개면 망한다? 정보 병목(Information Bottleneck)으로 규명한 멀티 에이전트(MAS)의 수학적 진실

* **논문 정보**: Yu et al., "When Do Multi-Agent Systems Help? An Information Bottleneck Perspective" (2026-07-17)
* **PDF 링크**: [https://arxiv.org/pdf/2607.16133v1](https://arxiv.org/pdf/2607.16133v1)
* **GitHub**: [https://github.com/divelab/MAS-SAS](https://github.com/divelab/MAS-SAS)

---

### ✨ [3줄 핵심 요약]
* **일관성 없는 성능의 실체**: 멀티 에이전트(MAS)는 고립된 컨텍스트 간의 '압축된 릴레이 메시지'로 통신하므로, 싱글 에이전트(SAS)와 달리 정보의 손실과 획득이 공존하는 구조입니다.
* **정보 병목(Information Bottleneck) 모델링**: MAS의 성능 이득은 무의미한 노이즈를 압축하는 '컨텍스트 제거 효과'와 핵심 정보를 잃어버리는 '릴레이 정보 손실' 사이의 트레이드오프 수식($G^{MAS} = H - \beta\Delta$)으로 완전히 정형화됩니다.
* **모델 수준($\beta$)에 따른 역전 현상**: 똑똑한 대형 모델(High $\beta$)일수록 스스로 복잡한 컨텍스트에서 정보를 정제할 수 있어 MAS의 압축 메커니즘이 도리어 손해를 끼치며, 약한 모델(Low $\beta$)일수록 압축된 가이드라인이 명확할 때 더 높은 성능 향상을 보입니다.

### 📖 [쉽게 풀어쓴 1분 핵심]
> **마라톤 회의를 혼자 적어가며 처리하는 것(SAS)과 여러 부서가 결재판으로 소통하는 것(MAS)의 차이**

* **싱글 에이전트(SAS)**는 **"천재 혼자 일을 다 하는 것"**과 같습니다. 하나의 메모장에 기획부터 코딩, 테스트 결과까지 모두 적어가며 일을 처리하죠. 하지만 메모장이 너무 길어지고 지저분해지면(컨텍스트 노이즈), 아무리 천재라도 정신이 혼미해져 치명적인 실수를 하기 시작합니다.
* **멀티 에이전트(MAS)**는 **"기획팀, 개발팀, QA팀"**이 나누어 일하는 것과 같습니다. 기획팀은 치열한 회의를 거쳐 핵심만 딱 정리한 **"기획서 한 장(릴레이 메시지, Relay)"**을 개발팀에 보냅니다. 이 방식은 개발팀이 쓸데없는 잡담에 시달리지 않게 도와줍니다.
* **여기서 아주 중요한 문제(Information Bottleneck)**가 생깁니다. 만약 기획서 요약본이 너무 짧아서 정작 개발팀에 꼭 필요한 핵심 비즈니스 요구사항(정보 손실, Relay Loss)이 잘려 나갔다면 어떻게 될까요?
  * **신입 개발자(약한 모델)**: 잡담 메일 수백 통을 다 읽다간 길을 잃어버리므로, 차라리 정보가 좀 빠졌더라도 깔끔하게 요약된 기획서 한 장이 오는 것을 훨씬 선호합니다.
  * **수석 개발자(강한 모델)**: 기획서 한 장에 적힌 수박 겉핥기식 요약보다, 차라리 기획팀이 주고받은 투박한 메일 전체를 다 파고들며 숨은 의도와 정확한 세부 지표를 읽어내는 것을 훨씬 잘하며, 요약본 전달 과정에서 유실된 정보 때문에 일이 망쳐지는 것을 극도로 싫어합니다.

이것이 바로 똑똑한 대형 언어 모델(LLM)이 등장할수록 멀티 에이전트 구조의 이점이 사라지거나 심지어 역전되는 수학적 원인입니다.

### 🎯 [배경 및 문제점]
> "무조건 에이전트를 쪼개면 잘 작동할 것이다"라는 무지성 분할 패러다임의 명확한 한계를 꼬집다.

최근 LLM 에이전트 생태계에서는 복잡한 문제를 풀기 위해 역할을 여러 개로 나누는 '멀티 에이전트 시스템(MAS)'이 대세였습니다. 코드 생성, 수학 추론 등 많은 벤치마크에서 MAS가 기존 싱글 에이전트(SAS)보다 좋은 성능을 보인다는 empirical(경험적) 연구들이 쏟아져 나왔습니다.

하지만 최근 들어 학계와 현업 모두에서 당혹스러운 반론들이 제기되기 시작했습니다:
> *"GPT-4o 같은 강력한 모델로 오니, 굳이 에이전트를 안 쪼개고 하나로 다 처리하는 게 성능이 더 좋은데요?"*
> *"멀티 에이전트가 성능이 더 좋은 건 단순히 에이전트 간 토론 과정을 거치면서 더 많은 연산량(Compute/Token Budget)을 썼기 때문 아닌가요?"*

그럼에도 불구하고 여태까지 **"정확히 어떤 물리적/수학적 메커니즘 때문에 특정 태스크에서는 MAS가 유리하고, 어떤 조건에서는 도리어 손해를 주는지"**를 시스템 정보 이론 관점에서 명쾌하게 규명한 연구는 없었습니다. 

이 논문은 바로 이 근본적인 물음, **"언제(When) 멀티 에이전트 아키텍처가 실제로 유효하고, 언제 손해가 발생하는가?"**를 통신 이론의 유서 깊은 체계인 **정보 병목(Information Bottleneck, IB)** 이론을 도입해 명확히 정리해 줍니다.

### 💡 [핵심 기술 및 아키텍처]
> 릴레이 메시지를 통한 컨텍스트 고립화와 정보 병목(Information Bottleneck) 수학식의 유도

이 연구는 에이전트가 정보를 관리하고 소통하는 흐름을 세 가지 통제된 아키텍처 프로토타입으로 격리하여 분석합니다.

#### 1. 대조 실험을 위한 세 가지 프로토타입 (Figure 2(a) 참조)
* **SAS (Single-Agent System)**: 한 명의 에이전트가 전체 맥락과 히스토리를 하나의 단일 공유 컨텍스트(Single shared context)에 모두 유지하며 순차적으로 문제를 풉니다.
* **SAS-contextflow**: 플래너가 설계한 하위 태스크 단계($X_1, X_2, \dots$)를 차례대로 밟아가되, 모든 컨텍스트 히스토리를 다른 에이전트에게 그대로 열어줍니다(Fully visible shared context). 즉, **태스크 분할 효과만을 MAS와 매칭시키고 정보 압축 효과는 제거**한 비교군입니다.
* **MAS (Multi-Agent System)**: 각 태스크 단계를 독립된 에이전트 워커가 처리하며, 각 워커는 고립된 로컬 컨텍스트를 갖습니다. 이전 워커는 오직 정제/요약된 **중계 메시지(Relay, $m_i$)**만 다음 워커에게 넘깁니다.

```
[SAS]          ---> [ (X1, o0, t1, a1, o1, t2, a2, o2) : 단일 대형 컨텍스트 ] ---> 최종 결과
[SAS-Flow]     ---> [ X1 단계 히스토리 ] ---> [ X1 + X2 단계 전체 히스토리 ] ---> 최종 결과 (전부 전달)
[MAS]          ---> [ X1 로컬메모리 ] --(압축 요약 m1)--> [ X2 로컬메모리 ] ---> 최종 결과 (요약만 전달)
```

#### 2. 무한 대역폭 등가성 (Infinite-Bandwidth Equivalence, Proposition 3.1)
논문은 이론적으로 매우 가치 있는 명제인 Proposition 3.1을 제안하고 증명합니다:
> 만약 에이전트가 다음 에이전트에 넘기는 릴레이 대역폭($B_{sys}$)이 무한대라면, 어떠한 SAS도 완벽하게 MAS로 모사할 수 있다.

즉, **MAS 고유의 본질적인 이점과 단점은 태스크의 분리 자체에서 오는 것이 아니라, 정보 유실을 강제하는 '제한된 릴레이 대역폭($B_{sys}$)'과 그에 따른 압축(Compression) 과정에서 발생**합니다.

#### 3. 정보 병목(IB) 공식으로 규명한 MAS Gain의 수학적 공식화
이전 워커의 전체 컨텍스트를 $M_i$, 다음 워커가 목표로 삼는 하위 태스크의 핵심 표적 변수를 $Y_{i+1}$, 그리고 그 과정에서 보낸 압축 중계 메시지를 $m_i$라고 할 때, 다음과 같은 **마르코프 연쇄 관계(Markov Relation)**가 성립합니다:

$$m_i \leftrightarrow M_i \leftrightarrow Y_{i+1} | X_{i+1}$$

여기서 최적의 중계 메시지($m_i$)는 다음의 **정보 병목(Information Bottleneck)** 목적 함수를 푸는 방식으로 정의됩니다:

$$\min_{p(m_i|M_i)} I(M_i; m_i) - \beta I(m_i; Y_{i+1} | X_{i+1})$$

* $I(M_i; m_i)$ : 상류 컨텍스트 정보를 얼마나 압축(줄였는가)했는지를 타겟팅합니다.
* $I(m_i; Y_{i+1} | X_{i+1})$ : 다운스트림 태스크에 꼭 필요한 유효 정보를 얼마나 살렸는지를 측정합니다.
* **$\beta$ (Effective $\beta$)**: **이 수식의 핵심 매개변수이며, 다운스트림 LLM의 역량(Capability)을 의미**합니다.

이 목적 함수를 바탕으로 싱글 에이전트(압축이 전무한 릴레이 $m_i = M_i$ 상태) 대비 **MAS 구조가 취하는 로컬 실질 이득(Local MAS Gain, $G_i^{MAS}$)**은 다음과 같이 아름답게 분해(Decomposition)됩니다:

$$G_i^{MAS} = \underbrace{H(M_i | m_i)}_{\text{upstream context reduction (이득 항)}} - \beta \times \underbrace{\Delta_i(m_i)}_{\text{relay information loss (손실 항)}}$$

* **이득 항 (컨텍스트 제거 효과)**: 이전 단계들의 지저분한 노이즈와 히스토리를 쳐내어 확보한 인지적 가벼움입니다.
* **손실 항 (릴레이 정보 손실)**: 정보를 요약하는 과정에서 필수적인 상세 정보가 소실되는 손실치입니다. 여기에 **모델의 역량($\beta$)이 가중치**로 작용합니다.

$$H(M_i | m_i) > \beta \Delta_i(m_i) \quad \text{일 때만 MAS가 SAS보다 유리하다!}$$

### 📊 [실험 결과 분석]
> 5개 벤치마크, 3가지 LLM 등급에 걸친 정교한 트레이드오프 증명

연구진은 가설 검증을 위해 세 가지 규모의 모델(`Qwen2.5-7B-Instruct`, `GPT-4o-mini`, `Qwen3.5-27B-AWQ`)을 바탕으로, 태스크의 결합도 및 릴레이 난이도(Relay Complexity $\delta$)에 따라 벤치마크를 정밀 설계하여 실험했습니다.

```
[복잡도 분류 (Relay Complexity δ)]
- δ ≈ 0 : ALFWorld(로봇 위치 정리), WideSearch(독립 병렬 웹검색), TravelPlanner-CS(로컬 상식 체크)
- δ > 0 : WebShop(속성 비교 쇼핑), TravelPlanner-HC(하드 전역 예산 제약 조건)
- δ ≫ 0 : WorkBench(메모리 참조 오피스 협업)
```

#### 1. 낮은 복잡도 영역 ($\delta \approx 0$) : MAS의 승리
정리 정돈이나 독립 병렬 웹 검색 같은 태스크에서는 다음 단계의 에이전트에 긴 대화 기록을 넘겨줄 필요 없이, "목표 탐색 완료: 가습기 위치는 테이블 1 위" 같은 아주 단순 명료한 릴레이($\delta \approx 0$)만으로 충분합니다.

* **실험 결과 (ALFWorld 성공률)**:
  * Qwen2.5-7B: **MAS (54.5%)** vs SAS-contextflow (17.2%) $\to$ **+19.4% 상승**
  * GPT-4o-mini: **MAS (55.2%)** vs SAS-contextflow (37.3%) $\to$ **+15.7% 상승**
  * Qwen3.5-27B: **MAS (89.5%)** vs SAS-contextflow (88.8%) $\to$ **+0.7% 상승**
* **해석**: 요약 메시지가 충분히 필요한 모든 것(Sufficiency)을 전달하므로 정보 손실 항($\Delta \approx 0$)이 0에 수렴합니다. 따라서 이득 항($H$)만 극대화되어 모든 성능 등급에서 MAS가 SAS를 일관되게 능가합니다. 똑똑한 모델일수록 어차피 긴 맥락을 스스로 잘 제어하므로 이득의 절대치(+0.7%)는 다소 감소합니다.

#### 2. 높은 복잡도 영역 ($\delta > 0$) : 파국적인 반전과 마이너스 게인
WorkBench(오피스 툴 제어)나 TravelPlanner-HC(전체 예산 하드 제약 조건 만족) 같이 이전 에이전트의 구체적인 실행 세부 내역(예: 정확한 일정 타임스탬프, API 호출 아이디, 세부 소요 예산 누적합)이 고스란히 하류 단계로 넘어가지 않으면 안 되는 복잡도 높은 환경입니다.

* **실험 결과 (TravelPlanner-HC 하드 제약 조건 만족률)**:
  * Qwen2.5-7B: MAS (1.7%) vs SAS-contextflow (0.0%) $\to$ 가벼운 향상
  * GPT-4o-mini: MAS (17.2%) vs SAS-contextflow (1.1%) $\to$ **MAS의 대폭 이득**
  * Qwen3.5-27B: MAS (25.6%) vs **SAS-contextflow (48.9%)** $\to$ **-23.3% 역전패!**

```
[TravelPlanner-HC 성능 비교 그래프]

  50% |                                      ● SAS-contextflow (48.9%)
      |
  25% |                     ▲ MAS (17.2%)    ▲ MAS (25.6%)
      |
   0% |  ● SAS (0.0%)
      +-------------------------------------------------------------
           Qwen2.5-7B          GPT-4o-mini      Qwen3.5-27B (강력한 모델)
```

* **해석 ($\beta$의 복수)**: 
  * 중소형 모델(`GPT-4o-mini`) 단계에서는 긴 전후맥락을 다 넘겨줬을 때 인지 부하로 인해 에이전트가 뻗어버리는 단점이 더 컸기 때문에 요약을 해주는 MAS 구조가 압도적으로 유리했습니다.
  * 하지만 강력한 모델인 `Qwen3.5-27B`에 이르자, 에이전트가 스스로 원본 원재료(Raw Shared Context)를 지저분한 히스토리 속에서 꺼내 분석하는 능력이 출중해집니다($\beta$ 값이 커짐). 이 시점부터는 **어설프게 릴레이 단계에서 압축되면서 전송되지 않은 예산 펑크 및 일정 불일치 누적 정보 손실 값($\beta\Delta$)이 도리어 노이즈 제거 이득($H$)을 아득히 압도**하게 되며, MAS가 처참하게 패배하게 됩니다.

### 🌍 [세상에 미치는 영향 & 실무 적용 사례]
> AI 에이전트 개발자여, 이제는 무지성 MAS 대신 '정보 결합도'를 계산하라.

엔터프라이즈 레벨의 복잡한 비즈니스 파이프라인이나 서비스에 AI 에이전트를 프로덕션으로 이식하고자 하는 리드 엔지니어 및 의사결정권자를 위해, 이 논문은 완벽한 아키텍처 의사결정 가이드를 제시합니다.

#### 1. 에이전트 설계 의사결정 트리 (Enterprise Architectural Decision Tree)

```
                       [해결하려는 태스크가 존재함]
                                     │
                 하위 단계로 넘어갈 때 꼭 필요한 정보가
             치밀하게 연결된 고결합 태스크(High-coupling)인가?
                                     │
                  ┌──────────────────┴──────────────────┐
               [ 아니오 (Low δ) ]                     [ 예 (High δ) ]
                  │                                     │
        각 워커에 저스펙 모델                     사용 중인 베이스 LLM이
    (GPT-4o-mini, Qwen-7B 등)을                충분히 강력하고 스마트한가?
      배치한 고효율 MAS 최적화                          │
                                             ┌──────────┴──────────┐
                                          [ 예 ]                [ 아니오 ]
                                             │                      │
                                      단일 대형 에이전트      도메인 데이터를 정교하게
                                    (SAS or Shared Context)   구조화된 Schema(JSON/DB)로
                                    기반 시스템 구성을 추천     변환하여 전달하는 MAS 구축
```

#### 2. 실무 적용 시나리오 A: 이커머스 맞춤형 추천 및 자동 결제 에이전트
* **Bad Practice (기존 MAS)**: '검색 에이전트'가 사용자 요청에 맞는 옷들을 쭉 검색하여 자연어 텍스트 요약 메시지(`m_i`)로 '결제 검증 에이전트'에게 대강 전달. 이 경우 정교한 가격 필터링, 실시간 재고 옵션, 사용자 할인 바우처 적용 조건들이 다 잘려 나가서 최종 결제 단계에서 하드 제약 조건 만족 실패.
* **Best Practice (논문 기반 개선)**: 릴레이 메시지 $m_i$의 대역폭을 보완하기 위해 자연어 요약 대신 **Schema-structured JSON State** 형태로 이전 단계의 Raw product metadata, DB query results를 그대로 매핑하여 전달. 즉, 손실 값 $\Delta_i \to 0$에 수렴하도록 통신 프로토콜을 통제하여 똑똑한 모델을 사용할 때도 MAS의 구조적 격리 이득을 그대로 흡수.

#### 3. 실무 적용 시나리오 B: 복잡한 사내 SQL 조회 및 보고서 자동화 워크플로우
* 사내 CRM 및 프로젝트 관리 대시보드(WorkBench 유사 도메인)를 넘나들며 조건부 쿼리를 짜는 에이전트 구축 시, 이전 단계 에이전트의 SQL 실행 원본 로그와 에러 트레이스 전체를 다음 검증 에이전트에 통째로 오픈하는 **Shared Context flow(SAS-contextflow)** 형태를 설계하십시오. 
* 어설픈 자연어 요약 릴레이는 고성능 LLM 단계에서 유실된 타임스탬프와 임시 유저 테이블 ID를 다시 찾지 못하게 해, 시스템 전체 성공률을 떨어뜨리는 지름길입니다.
