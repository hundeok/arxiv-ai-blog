# Loop the Loopies! 반복 연산(Looped) 트랜스포머의 한계를 깨고 올림피아드 골드 메달을 목에 걸다

안녕하세요! AI 기술 블로거이자 개발자 멘토인 **M**입니다. 

오늘 소개해 드릴 논문은 AI 아키텍처 설계의 패러다임을 바꿀 만한 엄청난 연구입니다. 바로 2026년 7월 17일 ArXiv에 공개된 **"Loop the Loopies!"**입니다. 

그동안 학계와 산업계에서는 "모델의 깊이(Depth)를 늘리기 위해 단순히 레이어를 더 많이 쌓는 것이 좋은가, 아니면 동일한 레이어를 반복(Looping)해서 재사용하는 것이 좋은가?"에 대해 치열한 논쟁이 있었습니다. 이 논문은 기존 루프형 트랜스포머의 치명적인 한계였던 **'동일 컴퓨팅 자원 대비 성능 열세'** 문제를 완벽하게 극복하고, 국제수학올림피아드(IMO)와 국제물리올림피아드(IPhO)에서 골드 메달 수준의 독보적인 추론 성적을 거둔 **Loopie** 시리즈를 공개했습니다.

지금부터 최고 수준의 기술 딥다이브와 함께, 이 혁신적인 아키텍처의 비밀을 파헤쳐 보겠습니다!

## 📖 [쉽게 풀어쓴 1분 핵심]
### 💡 아주 어려운 전공 서적을 읽을 때, 당신은 어떻게 읽으시나요?

> **어려운 책을 읽는 두 가지 방법**
> * **방법 A (기존의 모델-루프 방식):** 책 1페이지부터 100페이지까지 단숨에 쭉 읽고, 다시 처음으로 돌아와서 1페이지부터 100페이지까지 또 읽는다.
> * **방법 B (Loopie의 레이어-루프 방식):** 1페이지를 아주 깊게 두 번 읽고 완벽히 이해한 뒤, 2페이지로 넘어가서 또 두 번 읽는다.

기존의 반복형 AI(Model-loop)들은 방법 A처럼 동작했습니다. 전체 모델을 한 바퀴 다 돌린 후 처음부터 다시 돌렸죠. 이 방식은 앞 페이지 내용을 금방 까먹을 뿐만 아니라, 뇌(GPU 메모리)에 너무 많은 정보를 한꺼번에 얹어두어야 해서 머리가 아팠습니다(활성화 메모리 폭발).

반면, 이번에 발표된 **Loopie**는 방법 B(**Layer-loop**)를 사용합니다. 각 레이어(페이지)를 그 자리에서 연속으로 두 번 실행(반복 연산)하여 완벽하게 로컬 추론을 다듬은 뒤 다음 레이어로 넘어갑니다.

이렇게 하니 **"기억해 둬야 하는 페이지 수(저장된 활성화 메모리)"가 절반으로 줄어들어**, 남는 뇌 용량(메모리 여유 공간)으로 더 어려운 단어를 공부하거나(모델 가로폭 $D$ 확장) 한 번에 더 많은 책을 처리할 수 있게 된 것(배치 사이즈 2배 확대)입니다!

## 🎯 [배경 및 문제점]
### 1줄 요약: "루프 연산이 효율적이라는 환상 뒤에 숨겨진 뼈아픈 연산 비용 법칙(Compute-Accounting Problem)"

그동안 학계는 레이어의 가중치를 공유하여 반복 사용하는 루프형 트랜스포머(Looped Transformer)가 매개변수 효율성(Parameter Efficiency) 측면에서 훌륭하다고 칭송해 왔습니다. 하지만 이는 반쪽짜리 진실이었습니다. 현업과 거대 언어 모델(LLM) 사전 학습 환경에서 가장 중요한 제약 조건은 매개변수 개수가 아닌 **'총 사전 학습 컴퓨팅 예산(Pre-training Compute Budget, FLOPs)'**이기 때문입니다.

$$C \approx 6ND$$

모델을 $N$번 루프 돌린다는 것은, 사전 학습 단계에서 토큰당 연산량(FLOPs) 역시 $N$배로 정직하게 증가함을 의미합니다. 따라서 $N$번 루프를 도는 $2.6\text{B}$ 매개변수 모델은, 동일한 학습 비용을 소모하는 비루프형(Vanilla) $10.4\text{B}$ 모델($2.6\text{B} \times 4$)과 직접 경쟁해야 합니다. 불행히도 기존 연구들(Huginn, Ouro 등)은 이 고정된 컴퓨팅 자원(Fixed-FLOP) 비교에서 일반 트랜스포머의 성능을 뛰어넘지 못했습니다. 즉, **"차라리 그 돈(GPU 자원) 쓸 거면 레이어를 더 쌓는 게 낫다"**가 지배적인 정설이었습니다.

또한 기존 모델-루프(Model-loop) 방식은 다음과 같은 치명적인 한계가 있었습니다.
1. **인프라 비친화적 구조:** 전체 레이어 스택을 처음부터 끝까지 다 돌고 다시 처음 레이어로 피드백되는 루프 경계로 인해, 분산 환경(Pipeline Parallelism)에서 엄청난 통신 오버헤드와 파이프라인 버블(Idle State)이 발생합니다.
2. **기하급수적인 활성화 메모리(Activation Memory) 축적:** 역전파(Backpropagation)를 위해 모든 루프 단계의 중간 활성화 값을 GPU 메모리에 상주시켜야 하므로 메모리 부족(OOM) 현상이 극심해집니다.
3. **가중치 그래디언트 충돌(Gradient Conflict):** 한 레이어의 파라미터가 극단적으로 깊이가 다른 위치(예: 3번째 가용 깊이와 51번째 가용 깊이)에서 동시 학습되다 보니, 깊이별로 상이한 표현 학습(Representation Learning)의 요구사항 사이에서 학습 신호가 충돌합니다.

## 💡 [핵심 기술 및 아키텍처]
### 1줄 요약: "메모리 절약을 모델 성능 재투자로 전환하는 마법의 공식, Loopie Recipe"

Loopie 연구진은 이러한 패러독스를 깨부수기 위해 **Layer-loop 아키텍처**와 하드웨어 감지 기반의 **Loopie Recipe**를 제안했습니다.

### 1. Layer-loop: 깊이보다 로컬 반복에 집중하다

Loopie는 모델 전체를 돌리는 대신, 개별 Attention 및 MoE(Mixture-of-Experts) 블록 단위에서 로컬하게 2회 반복 연산을 수행하고 다음 블록으로 결과값을 전달합니다.

$$\text{Layer-loop: } L_1 \rightarrow L_1 \rightarrow L_2 \rightarrow L_2 \rightarrow L_3 \rightarrow L_3 \dots$$

```
[이전 레이어 출력] 
       │
       ▼
┌──────────────┐
│  Layer 1     │ ◄──┐ (1차 로컬 루프 반복)
└──────────────┘ ───┘
       │ (정제된 출력)
       ▼
┌──────────────┐
│  Layer 2     │ ◄──┐ (2차 로컬 루프 반복)
└──────────────┘ ───┘
       │ 
       ▼
[다음 레이어로 전달]
```

이 간단한 순서 변화는 엄청난 인프라적 이점을 가져옵니다.
* **실행 국소성(Execution Locality) 극대화:** 동일 레이어 가중치를 연속으로 사용하므로 하드웨어 가속기(GPU) 커널 내에서 파라미터 재사용 거리가 짧아집니다.
* **파이프라인 병렬화(PP) 최적화:** 모든 루프 연산이 해당 파이프라인 스테이지 안에서 완결되므로, 디바이스 간 활성화 값 전송 오버헤드나 파이프라인 스케줄링의 버블이 발생하지 않습니다.
* **자연스러운 파라미터 공유 패턴:** 가깝고 유사한 추론 단계(예: 가용 깊이 3과 4)에서 가중치를 공유하므로 이질적인 깊이 간의 그래디언트 충돌을 피할 수 있습니다.

### 2. 하드웨어 메모리를 쥐어짜는 Loopie Recipe

Loopie Recipe는 **"줄어든 활성화 메모리를 모델 물리적 크기(가로폭 $D$) 확장으로 환원하는"** 수학적인 스케일링 기법입니다.

활성화 메모리 $M_{act}$는 다음과 같은 관계를 갖습니다.

$$M_{act} \propto s \cdot b \cdot D \cdot L$$

*(여기서 $s$: 시퀀스 길이, $b$: 디바이스당 마이크로배치 사이즈, $D$: Hidden Dimension, $L$: 물리적으로 저장된 레이어 개수)*

Loopie는 저장 레이어 수($L$)를 절반으로 줄이고 루프 수($R=2$)를 설정하여 연산 효율을 보존하는 동시에, $L$이 절반이 됨에 따라 발생한 활성화 메모리 여유분을 활용하여 **마이크로배치 사이즈($b$)를 2배로 확장**합니다. 

이로 인해 발생한 연산 효율 향상분을 그대로 모델의 **가로폭(Hidden Dimension, $D$) 증가에 재투자**합니다. 이를 통해 실제 Megatron-LM 트레이닝 환경에서 단위 최적화 단계(Optimizer-step)당 벽시계 시간(Wall-clock Time)을 완벽히 맞추는 **'실질 연산 비용 매칭(Compute-Matched)'** 모델을 구축해냅니다.

| 모델 사양 (Model Specs) | Loopie-20B-A2B | Loopie-6B-A0.6B |
| :--- | :--- | :--- |
| **Layer-loop 반복 횟수 ($N$)** | 2 | 2 |
| **물리 저장 레이어 수 ($L$)** | 27 | 18 |
| **가로폭 (Hidden Size, $D$)** | 2304 | 1536 |
| **MoE 총 전문가 수 / 활성 수** | 128 / Top-8 | 128 / Top-8 |
| **총 파라미터 (Active 파라미터)** | 20B (2B) | 6B (0.6B) |

### 3. 혁신적인 SPT (Supervised Pre-Training) 기법

사후 학습(Post-training) 단계에서도 Loopie는 기존 SFT(Supervised Fine-Tuning)의 상식을 깹니다. 일반적인 SFT는 작은 배치 사이즈(32~128)와 짧은 시퀀스로 인해 쉽게 과적합(Overfitting)되거나 기존 지식을 망각(Catastrophic Forgetting)합니다. 

이를 방지하기 위해 Loopie 연구진은 **Supervised Pre-Training(SPT)** 기법을 도입했습니다.

* **사전 학습 스케일의 배치 최적화:** 글로벌 배치 사이즈 1024, 시퀀스 길이 128K, 총 2T(테라) 토큰 학습 예산을 적용합니다.
* **정밀 마스킹 손실 함수:** 프롬프트와 컨텍스트 토큰은 손실(Loss) 계산에서 완전히 제외하고, 오직 타겟 답변 토큰에 대해서만 크로스 엔트로피 손실을 계산합니다.

$$\mathcal{L}_{CE}(\theta; w) = -\frac{\sum_{i=1}^B \sum_{t=1}^{T_i} w_{i,t} \log p_\theta(z_{i,t} \mid z_{i,<t})}{\sum_{i=1}^B \sum_{t=1}^{T_i} w_{i,t}}$$

이 결과, 일반 지식(MMLU, ARC 등)의 파괴(망각)가 전혀 일어나지 않고, 추론 능력(AIME, OlympiadBench 등)과 상식적 기본기가 동시에 우상향하는 기염을 토했습니다.

> 3.5T 토큰 학습 모델이 25T 토큰 학습 모델과 대등하거나 더 나은 수치를 기록했다는 점은, **토큰 당 정보 학습 집약도가 최대 7배 이상 높다**는 것을 의미합니다.

### 3. 올림피아드 국가대표급 수학/물리 격파 현황
Loopie-20B-A2B는 **Generate-Verify-Refine**이라는 테스트 타임 스케일링(TTS) 기법을 사용했습니다. 한 문제당 64개의 후보 풀이를 만들고 비평(Critique) 프롬프트를 8회 이상 통과시켜 최상의 검증 값을 도출하는 루프 피드백 시스템입니다.

* **IMO 2025 (국제수학올림피아드):** 42점 만점에 **35점**을 획득하여, 2025년 대회 실제 컷오프 기준 **금메달(Gold Medal)**을 획득했습니다. (금메달 커트라인 35점)
* **IPhO (국제물리올림피아드):** 30점 만점에 **20.3점**을 달성하며, 금메달 기준선인 19.7점을 뛰어넘는 놀라운 결과를 만들어냈습니다.

## 🌍 [세상에 미치는 영향 & 실무 적용 사례]
### 1줄 요약: "고성능 GPU 품귀 시대, 인프라 비용을 극적으로 낮추는 새로운 차세대 에이전트 아키텍처"

현업 개발자나 기업 AI 아키텍트 입장에서 Loopie 기술은 단순한 연구를 넘어 강력한 실무 적용 시나리오를 제시합니다.

### 시나리오 1: 리소스 제한 기업의 고성능 도메인 특화 모델(SFT) 개발
* **문제 상황:** 사내 인하우스 엔지니어들이 A100/H100 GPU 개수가 한정되어 있어 대형 파라미터 모델을 풀 파인튜닝(Full Fine-tuning)하거나 큰 마이크로배치 사이즈(MBS)로 학습시킬 수 없어 OOM이 지속적으로 발생하는 상황.
* **실무 적용:** `megatron-loopie` 오픈소스를 활용하여 물리적인 저장 레이어 수는 줄이되, Layer-loop를 2회 돌리도록 변경합니다. 
* **효과:** 동일 컴퓨팅 FLOPs 대비 메모리 가용량이 늘어나기 때문에, **기존 GPU 인프라에서 마이크로배치 크기를 2배 이상 키우거나 컨텍스트 윈도우 길이를 극적으로 늘려(예: 32k -> 128k)** 인하우스 학습 효율을 획득할 수 있습니다.

### 시나리오 2: 초고성능 코딩 및 수학 에이전트 인프라 설계
* **문제 상황:** 복잡한 API 연동, 가상화 샌드박스 유닛 테스트 디버깅 등의 작업을 수행하는 AI 에이전트가 단발성 추론 오류로 인해 전체 워크플로우를 깨뜨리는 현상.
* **실무 적용:** 가볍고 압도적 추론 성능을 내는 Loopie-6B-A0.6B 혹은 20B 모델을 로컬 서빙 프레임워크(`vllm-loopie`)로 띄웁니다. 
* **효과:** 에이전트 설계 단계에 논문에서 증명한 '검증-반영 루프(Generate-Verify-Refine)' 파이프라인을 연동하면, 기업들은 비싼 상용 API(GPT-4o, Claude 3.5 Sonnet 등)를 지속 호출하지 않고도 **완벽히 격리된 온프레미스 환경에서 올림피아드 수준의 디버깅 및 복잡 로직 연산이 가능한 도메인 특화 에이전트**를 실현할 수 있습니다.
