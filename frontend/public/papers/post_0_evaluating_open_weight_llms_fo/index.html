실무적 코드 리팩토링을 위한 오픈 웨이트 LLM 평가

다양한 오픈 웨이트 LLM들이 실제 소프트웨어 개발 환경에서 코드 리팩토링 작업을 얼마나 효과적으로 수행하는지 검증합니다.

안녕하세요, 개발자 여러분! AI와 사이버 보안의 최전선에서 기술을 분석하고 성장을 돕는 멘토이자 기술 블로거입니다. 

오늘 소개해 드릴 논문은 자율주행차(Connected and Autonomous Vehicles, CAV) 보안의 핵심 패러다임을 바꿀 수 있는 매우 흥미로운 연구입니다. 바로 **"Evaluating Open-Weight LLMs for Generating Structured Threat Information for Autonomous Vehicle Vulnerabilities"**입니다. 

자율주행 차량에 숨겨진 보안 취약점(CVE) 텍스트를 AI를 이용해 기계가 즉시 읽을 수 있는 표준화된 보안 위협 정보 포맷(STIX)으로 변환하는 혁신적인 접근법을 다루고 있습니다. 이 논문을 통해 실무 보안 AI 아키텍처를 어떻게 설계해야 하는지 깊이 있게 파헤쳐 보겠습니다!

✨ [3줄 핵심 요약]

1. **최초의 자율주행 보안 데이터셋:** 자율주행 분야에 특화된 비정형 보안 취약점(CVE) 설명문과 표준 보안 포맷(STIX 2.1)을 매핑한 최초의 고품질 데이터셋 **`CAV-STIXGen`**을 구축했습니다.
2. **오픈웨이트 LLM 11종 비교:** 4B부터 120B에 이르는 11개의 오픈웨이트 모델을 대상으로 프롬프트 전략 및 멀티 에이전트 구조의 성능을 벤치마킹했습니다.
3. **뛰어난 성능 달성:** '동적 퓨샷(Dynamic Few-Shot)' 프롬프팅을 적용했을 때 개체 추출(SDO)에서 **F1 스코어 0.94**, 약점 매핑(CWE)에서 **0.99**라는 압도적인 정형화 성능을 입증했습니다.

📖 [쉽게 풀어쓴 1분 핵심]
> 💡 **1줄 요약:** 복잡한 자율주행 보안 일기를 컴퓨터가 즉시 이해할 수 있는 '레고 설계도'로 변환하는 AI 기술입니다.

자율주행 차량의 보안 취약점 정보(CVE)는 마치 사고 현장을 목격한 사람들이 각자의 말투로 적어 내려간 **"뒤죽박죽 메모장"**과 같습니다. 어떤 사람은 엔진 제어기(ECU)가 멈췄다고 하고, 어떤 사람은 원격에서 해킹이 된다고 제각각 설명하죠.

이 문제를 해결하기 위해 보안 전문가들은 이 메모를 바탕으로 정밀한 **"레고 설계도(STIX 포맷)"**를 그리려고 합니다. 이 설계도에는 다음과 같은 명확한 조각들이 필요합니다.
* **빨간 블록:** 해킹당한 부품 (예: 인포테인먼트 시스템)
* **파란 블록:** 공격자가 쓴 나쁜 짓 (예: 서비스 거부 공격)
* **연결 핀:** 빨간 블록과 파란 블록의 관계

**이 논문에서 구현한 AI는 바로 이 '메모장'을 읽고, 한 치의 오차도 없이 완벽한 '레고 설계도(JSON 코드)'로 조립해 주는 똑똑한 로봇 조립사입니다.** 덕분에 보안 전문가들이 수작업으로 수십 시간씩 설계도를 그리던 작업을 AI가 눈 깜짝할 사이에 끝마칠 수 있게 되었습니다.

🎯 [배경 및 문제점]
> 💡 **1줄 요약:** 비정형 보안 데이터(CVE)는 기계가 읽을 수 없어 실시간 보안 관제 및 자동 방어가 불가능했습니다.

1. 자율주행 보안의 특수성과 취약성
자율주행차(CAV)는 라이다(LiDAR), 레이더, 카메라, 전자제어장치(ECU), 차량 내 인포테인먼트(IVI), V2X(Vehicle-to-Everything) 통신 등이 복잡하게 얽힌 하나의 거대한 '바퀴 달린 컴퓨터'입니다. 2024년 기아(Kia) 차량이 번호판만으로 원격 제어되거나, 2025년 스바루(Subaru)의 스타링크 취약점으로 위치 추적이 가능했던 사례처럼 자율주행 보안 위협은 현실의 생명과 직결됩니다.

2. 비정형 데이터의 한계
이러한 취약점들은 CVE(Common Vulnerabilities and Exposures) 데이터베이스에 **자연어 텍스트**로 등록됩니다. 하지만 보안 관제(SOC) 및 자동화 대응 시스템이 이를 활용해 방어 장비를 실시간 제어하려면, 데이터가 **STIX 2.1(Structured Threat Information Expression)과 같은 구조화된 그래프 데이터**로 변환되어야 합니다.

```
[비정형 자연어 (CVE-2023-28901)]
"Skoda 차량 클라우드의 접근 제어 결함으로 원격 공격자가 차대번호(VIN)를 통해 사용자 정보에 접근할 수 있음."
                      ⬇️ (LLM의 정형화 작업 필요)
[구조화 온톨로지 (STIX 2.1)]
{ "type": "vulnerability", "id": "vuln--1", "name": "Broken Access Control" },
{ "type": "relationship", "relationship_type": "exploits", "source_ref": "attacker", "target_ref": "vuln--1" }
```

3. 기존 연구 및 LLM 적용의 한계점
1. **특화 데이터셋 부재:** 일반 IT 시스템 대상의 CTI(Cyber Threat Intelligence) 데이터셋은 많지만, 자율주행 전용 하드웨어 및 통신 특성을 반영한 CVE-to-STIX 벤치마크는 없었습니다.
2. **LLM의 환각(Hallucination):** 가이드 없이 LLM에게 구조화 데이터 생성을 맡기면 존재하지 않는 STIX 객체 타입을 지어내거나(Hallucinated attribute), JSON 문법 오류를 내고, 관계 정의(SRO)를 잘못 설정하는 고질적인 문제가 있었습니다.

---

💡 [핵심 기술 및 아키텍처]
> 💡 **1줄 요약:** 특화 벤치마크 'CAV-STIXGen'을 최초 설계하고, '동적 퓨샷' 및 '멀티 에이전트' 파이프라인으로 정확도를 극대화했습니다.

이 연구는 단순히 LLM에게 번역을 시키는 것을 넘어, 엄격한 제어 아키텍처를 도입했습니다.

1. CAV-STIXGen 데이터셋 구축 흐름
논문은 엄밀한 기준을 거쳐 골드 스타 표준(Gold Standard) 데이터셋을 구축했습니다.
1. **키워드 필터링:** 자율주행 제조사(Tesla, BMW 등), V2X, CAN 버스 등 도메인 키워드를 도출해 NVD에서 445개 후보 추출.
2. **수동 필터링 (Relevance Filtering):** 일반 웹사이트나 영업 플랫폼 취약점은 제외하고 실제 차량 연동 시스템 취약점 183개 선별 (평가자 간 일치도 Cohen’s $\kappa = 94\%$).
3. **수동 주석 및 검증:** 각 CVE에 대해 STIX 도메인 객체(SDO), 관계 객체(SRO), CWE, MITRE ATT&CK 기술 ID를 직접 매핑하고 정합성이 검증된 JSON 템플릿으로 변환.

2. 세 가지 프롬프팅 전략 (Prompting Strategies)
LLM의 문맥 이해도를 극대화하기 위해 세 단계의 프롬프팅 아키텍처를 비교 분석했습니다.

* **A. 무맥락 프롬프팅 (Contextless Prompting):** 단순 지시와 함께 CVE 텍스트만 주고 STIX JSON 생성을 요구합니다.
* **B. 가이드 기반 프롬프팅 (STIX-Guided Prompting):** 19가지 STIX 객체 사양 규칙, CWE 체계 매핑 가이드, MITRE ATT&CK 기술 정의 등을 규칙(Rule)으로 제공합니다.
* **C. 동적 퓨샷 프롬프팅 (Dynamic Few-Shot Prompting):** 입력된 CVE의 텍스트와 의미론적(Semantic) 임베딩 유사도가 가장 높은 5개의 실제 정답 케이스(CWE, Attack Pattern 매핑 정보 포함)를 벡터 검색으로 찾아 프롬프트 컨텍스트에 동적으로 주입합니다.

```
[동적 퓨샷 검색 구조]
새로운 CVE 텍스트 입력 ➡️ 임베딩 모델(유사도 연산) ➡️ CAV-STIXGen DB ➡️ 가장 유사한 예제 5개 자동 추출 ➡️ LLM 프롬프트에 조립 ➡️ 고품질 STIX JSON 생성
```

3. 멀티 에이전트 협업 아키텍처 (Multi-Agent Setup)
하나의 거대 모델이 모든 태스크를 처리할 때 생기는 병목을 해결하기 위해, 역할을 세분화한 **멀티 에이전트 시스템**을 구현했습니다.
* **Entity Extractor Agent:** CVE 문맥에서 주요 인프라, 위협 행위자 등 SDO 개체를 우선 추출.
* **Mapper Agent:** 추출된 개체 기반으로 CWE 약점 및 MITRE ATT&CK 표준 기술 매핑.
* **Relationship Builder Agent:** 개체와 매핑 정보를 연결해 완벽한 SRO(관계) 정의.
* **Validator Agent:** 최종 출력 JSON의 문법적 오류와 온톨로지 규칙 위배 여부를 실시간 검사 및 피드백 루프 수행.

---

📊 [실험 결과 분석]
> 💡 **1줄 요약:** 동적 퓨샷 프롬프팅이 성능을 하드캐리했으며, 개체 추출 F1 스코어 0.94 및 CWE 매핑 0.99의 압도적 결과를 도출했습니다.

이 연구는 총 11종의 오픈웨이트 모델(Gemma-3/4, Phi-4, Qwen-3.5/Coder, Lily-7B, CyberSec-Qwen, Codestral-22B, GPT-OSS, LLaMA-3.3-70B 등)을 비교했습니다.

1. 단일 모델 벤치마크 결과 (F1 스코어)

| 모델명 (Model) | 프롬프트 전략 | SDO (개체) F1 | SRO (관계) F1 | CWE F1 | ATT&CK (Match@1) |
| :--- | :--- | :---: | :---: | :---: | :---: |
| **Phi-4** | Dynamic Few-Shot | **0.94** | 0.55 | 0.94 | 0.53 |
| **Qwen3-Coder-30B** | Dynamic Few-Shot | **0.94** | **0.63** | **0.99** | 0.67 |
| **LLaMA-3.3-70B** | Dynamic Few-Shot | **0.94** | 0.51 | **0.99** | 0.61 |
| **Codestral-22B** | Dynamic Few-Shot | 0.93 | 0.54 | **0.98** | 0.45 |
| **Gemma-4-31B** | Dynamic Few-Shot | 0.80 | 0.61 | 0.95 | **0.68** |

* **SDO(개체 추출) 성능:** 대다수의 고성능 오픈웨이트 모델이 **F1 0.94** 수준의 놀라운 정밀도를 보여주었습니다. 가이드가 없을 때(Contextless)는 0.5~0.6 수준에 머물렀으나 동적 퓨샷이 적용되자 비약적으로 도약했습니다.
* **SRO(관계 연결)의 장벽:** 그래프 연결 상태를 추론하는 SRO는 상대적으로 까다로운 작업이었습니다. 코드 역량이 뛰어난 **Qwen3-Coder-30B 모델이 F1 0.63**으로 가장 우수한 능력을 발휘했습니다.
* **CWE 매핑:** 최적의 프롬프팅 하에 **F1 0.99**를 기록하며, 인간 전문가 수준의 약점 분류 정확도를 확보했습니다.

2. 멀티 에이전트 vs 단일 모델 (Dynamic Few-Shot)
* **Gemma-4-31B 멀티 에이전트:** SDO 추출 F1 **0.91** 달성.
* **Codestral-22B 멀티 에이전트:** SRO 추출 F1 **0.43** 달성.
* *인사이트:* 멀티 에이전트 구조가 특정 태스크에서 구조적 일관성을 확보하는 데는 도움을 주었으나, 오버헤드와 모델 간 통신 누수로 인해 잘 튜닝된 단일 모델의 '동적 퓨샷 프롬프팅' 성능을 완전히 뛰어넘지는 못했습니다.

3. 자율주행 보안 도메인 취약점 분석 (FP-Growth 분석)
연구진은 확보한 데이터셋을 바탕으로 빈발 아이템 세트 마이닝(FP-Growth)을 수행해 자율주행차 취약점의 숨겨진 트렌드를 밝혀냈습니다.

* **최다 발생 CWE 약점:** **CWE-787(Out-of-bounds Write, 20회)**, **CWE-284(Improper Access Control, 14회)**. 자율주행 하드웨어 제어 시 메모리 안전성 문제와 차량 클라우드 통신 시 인가 처리가 취약하다는 점이 증명되었습니다.
* **최다 발생 MITRE ATT&CK 기술:** **T1499(Endpoint Denial of Service, 48회)**, **T1203(Exploitation for Client Execution, 42회)**.
* **가장 빈번한 동시 발생 패턴:** **T1203(클라이언트 실행 취약점 공략) ➡️ T1499(엔드포인트 DOS 공격 유발)** 쌍이 총 10개 CVE에서 동시에 발견되었습니다. 즉, 자율주행 장치에 원격 코드를 실행시켜 차량 시스템 자체를 다운시키는 연쇄 공격 패턴이 지배적이라는 뜻입니다.

🌍 [세상에 미치는 영향 & 실무 적용 사례]
> 💡 **1줄 요약:** 차량 보안 관제 센터(VSOC)의 인적 개입 비용을 90% 이상 절감하고 수작업 분석 한계를 극복합니다.

이 기술은 단순 연구실 시뮬레이션에 머무르지 않고 기업형 솔루션에 즉시 적용할 수 있는 구체적인 가치를 제공합니다.

실무 적용 시나리오 1: VSOC(차량 보안 관제 센터)의 실시간 CTI 파이프라인 자동화
자동차 제조사(OEM)나 모빌리티 서비스 기업은 매일 업데이트되는 신규 CVE 정보를 실시간 모니터링해야 합니다.

1. **설계 구조:** 기업 내부 가상 서버에 오픈웨이트 모델인 `Qwen3-Coder-30B` 또는 `Phi-4`를 호스팅합니다. (vLLM 인프라 활용)
2. **파이프라인 구축:**
   * NVD API에서 실시간으로 차량 보안 키워드가 섞인 신규 CVE 수집.
   * 구축된 **`CAV-STIXGen` 기반 벡터 저장소**에서 임베딩 유사도로 예제 5개 실시간 검색.
   * 동적 퓨샷 프롬프트에 밀어 넣어 1초 만에 표준 STIX JSON 생성.
   * 변환된 JSON 데이터를 사내 위협 정보 플랫폼(OpenCTI, MISP 등)으로 즉시 전송해 방어 시나리오 및 위협 그래프 자동 생성.

실무 적용 시나리오 2: 자율주행 차량 위협 모델링 및 가상 공격 시뮬레이션
보안 설계자가 신규 차량 모델을 개발할 때, 개발 중인 하드웨어 장치(예: MCU, Telematics Unit)의 알려진 약점(CWE)을 기반으로 예상되는 공격 시나리오를 시뮬레이션할 수 있습니다.

* **동작 방식:** 설계 단계에서 발견된 CWE 약점 코드를 시스템에 입력하면, AI가 매핑 네트워크를 기반으로 어떤 MITRE ATT&CK 전술(예: T1210 - 원격 서비스 익스플로잇 ➡️ T1499 - 차량 오동작 도스 공격)로 이어질 수 있는지 위협 트리(Threat Tree)를 즉시 시각화해 줍니다. 

멘토의 실무 팁 💡
> "사이버 보안 영역에서는 데이터 기밀성이 최우선입니다. GPT-4와 같은 상용 API를 사용할 경우 차량 설계 데이터나 취약점 원문이 외부로 노출되는 리스크가 발생합니다. 본 연구에서 검증된 **Codestral-22B**나 **Phi-4** 같은 오픈웨이트 모델을 온프레미스(On-Premise) GPU 장비에 로컬로 구축하여 사용하면, **데이터 유출 걱정 없이 완벽한 독점적 보안 인텔리전스 시스템**을 내재화할 수 있습니다."
